# Домашнее задание: Внедрение мониторинга в проект (Observability)

## Контекст
Бизнес-логика вашего сервиса уже реализована. Теперь ваша задача — сделать систему «прозрачной». Нужно внедрить логирование и метрики, чтобы любое действие пользователя было заметно на графиках, а любую ошибку можно было расшифровать по логам за считанные минуты.

Работа разделена на 4 равные части. Каждый участник реализует мониторинг в своем функциональном модуле и выполняет одну общую техническую задачу.

---

## Общие требования для всех
В своем модуле (функциональном блоке) каждый участник должен внедрить:

1.  **INFO логи**: фиксация значимых событий (например, *«Заказ №123 оплачен»*, *«Пользователь вошел в систему»*).
2.  **DEBUG логи**: фиксация технических деталей для отладки (значения переменных, результаты вычислений, логика ветвления `if-else`).
3.  **Метрики**: добавить минимум один `Counter` (счетчик) на важное событие (например, количество созданных записей или успешных операций).

---

## Распределение задач по частям

### Часть 1
*   **Инфраструктура**: Подготовить файл `docker-compose.yml` для запуска **Prometheus** и **Grafana** в контейнерах.
*   **Настройка логов**: Сконфигурировать файл `logback-spring.xml`. Логи должны выводиться в консоль (цветной текст) и в файл `logs/app.json` в формате **JSON** (используя LogstashEncoder).
*   **Реализация**: Внедрить INFO/DEBUG логи и метрики в **функциональный модуль №1**.

### Часть 2
*   **Трассировка (TraceID)**: Реализовать механизм **MDC** (через Filter или Interceptor), который присваивает каждому входящему запросу уникальный ID. Этот ID должен автоматически проставляться во всех строках лога для этого запроса.
*   **Обработка ошибок**: Настроить глобальный обработчик исключений (`@RestControllerAdvice`). Он должен перехватывать ошибки, записывать их в лог вместе с `traceId` и возвращать пользователю JSON с описанием проблемы и этим же ID.
*   **Реализация**: Внедрить INFO/DEBUG логи и метрики в **функциональный модуль №2**.

### Часть 3
*   **Сбор данных**: Настроить файл `prometheus.yml`, чтобы Prometheus начал забирать метрики из вашего Spring Boot приложения (эндпоинт `/actuator/prometheus`).
*   **Бизнес-панель**: Создать в Grafana дашборд с **бизнес-метриками** проекта (например: сколько всего заказов создано, сколько пользователей зарегистрировано, топ категорий).
*   **Реализация**: Внедрить INFO/DEBUG логи и метрики в **функциональный модуль №3**.

### Часть 4
*   **Техническая панель (RED)**: Создать в Grafana графики для мониторинга «здоровья» API на основе стандартных метрик Spring Boot:
    *   **Rate**: количество запросов в секунду (RPS).
    *   **Errors**: количество и процент ошибок (статусы 4xx и 5xx).
    *   **Duration**: время ответа сервера (используя метрику `http_server_requests_seconds`).
*   **Реализация**: Внедрить INFO/DEBUG логи и метрики в **функциональный модуль №4**.

---

## Требования к логированию (Важно!)

Для того чтобы логи были полезными, соблюдайте правила:
*   **INFO**: только важные вехи. Пример: *«Пользователь с id = 1 сформировал заказ»*
*   **DEBUG**: подробный путь данных. Пример: *«Скидка 10% применена к заказу #55. Итоговая сумма: 900»*.
*   **Включение DEBUG**: Не забудьте включить ваш пакет в `application.yaml`, иначе DEBUG-логи не отобразятся:
    ```yaml
    logging:
      level:
        ru.your.project.package: DEBUG
    ```

## Что должно получиться в результате

1.  **Код**: Проект с настроенными логами и метриками во всех модулях.
2.  **Логи**: Файл `app.json`, в котором по одному `traceId` можно отследить весь путь конкретного запроса.
3.  **Дашборд**: Работающая Grafana с техническими (RED) и бизнес-графиками.
4.  **Дашборд Grafana в JSON**: Дашборд Grafana должен быть экспортирован в формат **JSON** и сохранен в репозитории (кнопка *Share -> Export -> Save to file*).

## Инструкция по проверке
1.  Запустить проект и выполнить несколько HTTP-запросов к разным частям сервиса.
2.  Открыть Grafana и импортировать JSON-файл вашего дашборда.
3.  Специально вызвать ошибку в приложении (например, передав некорректные данные).
4.  Найти всплеск ошибок на техническом графике.
5.  Взять `traceId` из ответа сервера и найти подробные **DEBUG**-логи этой ошибки в файле `app.json`.
